## 一、Agent 架构相关

### 1. REACT、COT、Action 三种 Agent 架构的区别是什么？各自适用于什么场景？

**COT (Chain of Thought)**
- **核心思想**：思维链，让模型在给出答案前先进行逐步推理
- **特点**：通过"让我们一步步思考"等方式激发模型的推理能力
- **适用场景**：
  - 数学计算题
  - 逻辑推理题
  - 需要多步推导的复杂问题

**REACT (Reasoning + Acting)**
- **核心思想**：推理与行动相结合，模型可以调用外部工具/API
- **特点**：交替进行推理和行动，通过观察环境反馈来调整策略
- **适用场景**：
  - 需要查询外部信息的任务
  - 需要调用 API 或工具的场景
  - 复杂的多步骤任务（如网页浏览、代码执行）

**Action 模型**
- **核心思想**：专注于执行具体动作，强调与环境的交互
- **特点**：更注重行动的准确性和效率，通常有预定义的动作空间
- **适用场景**：
  - 游戏 AI
  - 机器人控制
  - 自动化任务执行

**三者区别总结**：
| 架构 | 核心能力 | 是否需要工具 | 典型应用 |
|------|---------|-------------|---------|
| COT | 纯推理 | 否 | 数学题、逻辑题 |
| REACT | 推理 + 行动 | 是 | 复杂任务规划 |
| Action | 行动执行 | 是 | 自动化、控制 |

---

### 2. REACT 模型适合解决什么问题？Action 模型的特点是什么？

**REACT 模型适合解决的问题**：
- 需要多步推理和信息收集的任务
- 需要调用外部工具/API 的场景（如搜索、计算、代码执行）
- 开放域问答（需要查资料验证）
- 复杂的问题解决（如规划旅行、写代码并调试）

**Action 模型的特点**：
- 有明确定义的动作空间
- 强调与环境的实时交互
- 通常使用强化学习进行训练
- 注重行动的准确性和效率
- 适合序列决策问题

## 二、Java 相关

### 3. Java 运行时内存区域（线程私有与共享）是如何划分的？程序计数器的作用是什么？

**Java 内存区域划分**：

**线程私有（每个线程独立）**：
1. **程序计数器 (Program Counter Register)**
   - 记录当前线程执行的字节码指令地址
   - 唯一不会 OOM 的区域
2. **Java 虚拟机栈 (Java Virtual Machine Stack)**
   - 存储栈帧（局部变量表、操作数栈、动态链接、方法出口）
   - 每个方法调用对应一个栈帧
3. **本地方法栈 (Native Method Stack)**
   - 为 Native 方法服务

**线程共享（所有线程共享）**：
1. **堆 (Heap)**
   - 存放对象实例
   - 垃圾回收的主要区域
   - 可细分为：新生代（Eden、Survivor）、老年代
2. **方法区 (Method Area) / 元空间 (Metaspace)**
   - 存储类信息、常量、静态变量
   - JDK8 后用元空间替代永久代

**程序计数器的作用**：
- 记录当前线程执行的字节码指令的行号指示器
- 分支、循环、跳转、异常处理、线程恢复都依赖它
- 多线程环境下，线程切换后能知道继续执行的位置

---

### 4. Java 内存模型（JMM）中线程内存与主内存如何同步？

**JMM 核心概念**：
- 每个线程有自己的工作内存（本地内存）
- 所有变量存储在主内存中
- 线程不能直接访问主内存，必须通过工作内存

**同步机制（8 个原子操作）**：

**主内存 → 工作内存**：
1. `read`：从主内存读取变量
2. `load`：载入到工作内存

**工作内存 → 主内存**：
3. `use`：使用变量值
4. `assign`：赋值操作
5. `store`：存储到主内存
6. `write`：写入主内存

**锁定机制**：
7. `lock`：锁定主内存变量
8. `unlock`：解锁

**volatile 关键字保证**：
- 可见性：一个线程修改后，其他线程立即可见
- 有序性：禁止指令重排序
- 不保证原子性

**synchronized 保证**：
- 原子性
- 可见性
- 有序性

### 5. ConcurrentHashMap 与 HashMap 的区别是什么？

| 特性 | HashMap | ConcurrentHashMap |
|------|---------|-------------------|
| 线程安全 | 否 | 是 |
| 锁机制 | 无锁 | JDK7:分段锁；JDK8:CAS+synchronized |
| null 键值 | 允许 null 键和 null 值 | 不允许 null 键和 null 值 |
| 迭代器 | 非安全（fail-fast） | 安全（弱一致性） |
| 性能 | 单线程高 | 多线程高 |

**JDK8 ConcurrentHashMap 实现**：
- 采用数组 + 链表 + 红黑树（同 HashMap）
- 锁粒度：桶（Node）级别，而非整个段
- 使用 CAS 保证插入操作的原子性
- 链表长度≥8 且数组长度≥64 时转为红黑树

### 6. HashMap 的扩容机制是怎样的？

**扩容条件**：
- 元素个数 > 容量 × 负载因子（默认 16 × 0.75 = 12）

**扩容过程**：
1. 创建新数组，容量为原来的 2 倍
2. 重新计算每个元素的 hash 值
3. 将元素重新分配到新数组（rehash）

**JDK8 优化**：
- 不需要重新计算 hash
- 根据 `(e.hash & oldCap) == 0` 判断元素位置
  - 结果为 0：保持在原索引位置
  - 结果为 1：移动到原索引 + oldCap 位置

**注意事项**：
- 扩容是性能开销较大的操作
- 初始化时指定合适容量可减少扩容次数
- 负载因子越大，空间利用率越高，但冲突概率越大

### 7. 线程和进程的区别是什么？

| 特性 | 进程 | 线程 |
|------|------|------|
| 定义 | 资源分配的最小单位 | CPU 调度的最小单位 |
| 内存空间 | 独立的地址空间 | 共享进程内存 |
| 通信方式 | IPC（管道、消息队列、共享内存等） | 直接读写共享数据 |
| 创建开销 | 大 | 小 |
| 切换开销 | 大 | 小 |
| 独立性 | 高，一个进程崩溃不影响其他 | 低，一个线程崩溃可能导致整个进程崩溃 |
| 包含关系 | 一个进程可包含多个线程 | 线程属于进程 |

**形象比喻**：
- 进程 = 工厂
- 线程 = 工厂里的工人
- 工厂有独立厂房（内存空间），工人共享厂房资源

### 8. CPU 或内存使用异常时，如何进行线上排查？

**CPU 使用率过高排查**：

```bash
# 1. 定位高 CPU 进程
top

# 2. 查看进程内线程 CPU 占用
top -Hp <pid>

# 3. 将线程 ID 转为 16 进制
printf "%x\n" <tid>

# 4. 导出线程栈
jstack <pid> | grep <16 进制 tid> -A 50

# 5. 分析栈信息，定位问题代码
```

**常见原因**：
- 死循环
- 频繁 GC
- 复杂计算
- 线程池配置不当

**内存使用异常排查**：

```bash
# 1. 查看 JVM 内存使用情况
jstat -gc <pid> 1000

# 2. 导出堆 dump
jmap -dump:format=b,file=heap.hprof <pid>

# 3. 分析堆 dump（MAT、JVisualVM）

# 4. 查看对象占用
jmap -histo <pid> | head -50
```

**常见原因**：
- 内存泄漏
- 大对象过多
- 内存配置不当

## 三、数据库相关

### 9. 数据库查询变慢时如何定位问题？

**排查步骤**：

1. **开启慢查询日志**
```sql
-- 查看慢查询配置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 开启慢查询
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;
```

2. **分析慢查询日志**
```bash
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log
```

3. **使用 EXPLAIN 分析执行计划**
```sql
EXPLAIN SELECT * FROM table WHERE ...;
```
关注：type、key、rows、Extra

4. **查看锁等待**
```sql
SHOW PROCESSLIST;
SELECT * FROM information_schema.INNODB_TRX;
SELECT * FROM information_schema.INNODB_LOCKS;
```

5. **检查索引使用情况**
```sql
SHOW INDEX FROM table_name;
```

**常见原因**：
- 缺少索引或索引失效
- 锁等待/死锁
- 查询语句不合理
- 表数据量过大
- 服务器资源不足

### 10. MySQL 中有哪些索引类型？它们的区别是什么？

**按数据结构分类**：
1. **B+ 树索引**：最常用，适合范围查询
2. **Hash 索引**：等值查询快，不支持范围
3. **Full-Text 全文索引**：文本搜索
4. **R-Tree 空间索引**：地理空间数据

**按物理存储分类**：
1. **聚簇索引（主键索引）**
   - 数据与索引存储在一起
   - 一张表只有一个
   - InnoDB 的主键就是聚簇索引

2. **非聚簇索引（二级索引）**
   - 索引与数据分离存储
   - 叶子节点存储主键值
   - 需要回表查询

**按逻辑特性分类**：
1. **普通索引**：无限制
2. **唯一索引**：值唯一，允许 NULL
3. **主键索引**：唯一且非 NULL
4. **联合索引**：多列组合
5. **覆盖索引**：查询列都在索引中，无需回表

**最左前缀原则**：
- 联合索引 (a,b,c)，查询条件必须从 a 开始才能使用索引
- (a=1 AND b=2) 可用索引
- (b=2 AND c=3) 不可用索引

## 四、网络相关

### 11. TCP 中客户端 ESTABLISHED 而服务端 LISTEN 可能是什么问题？

**可能原因**：

1. **服务端未调用 accept()**
   - 服务端 listen 后， backlog 队列已满
   - 新连接无法建立

2. **防火墙/安全组拦截**
   - SYN 包到达，但后续包被拦截
   - 连接卡在 SYN_RECV 状态

3. **服务端处理慢**
   - accept 队列满，新连接被丢弃
   - 客户端显示 ESTABLISHED 但实际未完全建立

4. **网络延迟或丢包**
   - ACK 包丢失，服务端未收到
   - 客户端已收到 SYN-ACK，发送 ACK

**排查方法**：
```bash
# 查看连接状态
netstat -an | grep ESTABLISHED
ss -tan | grep ESTABLISHED

# 查看 SYN 包
tcpdump -i any port <port> -n
```

### 12. TCP 四次挥手的过程是怎样的？

**挥手过程**：

```
客户端                    服务端
  |                        |
  |-----FIN (seq=u)------->|  1. 客户端发送 FIN，进入 FIN_WAIT_1
  |                        |
  |<----ACK (ack=u+1)------|  2. 服务端确认，客户端进入 FIN_WAIT_2
  |                        |     服务端进入 CLOSE_WAIT
  |                        |
  |                        |  3. 服务端发送完数据后
  |<----FIN (seq=v)--------|     发送 FIN，进入 LAST_ACK
  |                        |
  |-----ACK (ack=v+1)----->|  4. 客户端确认，进入 TIME_WAIT
  |                        |     服务端进入 CLOSED
  |                        |
  |  (2MSL 后关闭)          |
```

**状态说明**：
- **FIN_WAIT_1**：等待对方 ACK
- **FIN_WAIT_2**：等待对方 FIN
- **CLOSE_WAIT**：等待本方应用关闭
- **LAST_ACK**：等待最后 ACK
- **TIME_WAIT**：等待 2MSL，确保对方收到 ACK

**为什么需要 TIME_WAIT**：
1. 确保最后一个 ACK 能到达服务端
2. 防止旧连接的重复报文影响新连接

### 13. TCP 连接异常断开（如断电）会发生什么？如何判断连接是否存活？

**异常断开的情况**：

1. **正常断电/断网**
   - 没有发送 FIN 包
   - 对方连接处于"半开"状态
   - 直到发送数据时才会发现

2. **检测机制**：

**心跳包**：
```
应用层定期发送心跳消息
对方收到后回复
超时未回复则认为连接断开
```

**TCP Keepalive**：
```bash
# 查看系统配置
cat /proc/sys/net/ipv4/tcp_keepalive_time  # 默认 7200 秒
cat /proc/sys/net/ipv4/tcp_keepalive_intvl # 探测间隔
cat /proc/sys/net/ipv4/tcp_keepalive_probes # 探测次数
```

**应用层超时**：
- 设置读写超时时间
- Socket 层设置 SO_RCVTIMEO/SO_SNDTIMEO

**判断连接存活的方法**：
1. 应用层心跳（推荐）
2. TCP Keepalive（默认关闭）
3. 发送数据探测（有副作用）

## 五、算法题

### LeetCode 79. 单词搜索

**题目**：
给定一个 m x n 二维字符网格 board 和一个字符串单词 word。如果 word 存在于网格中，返回 true；否则，返回 false。

单词必须按照字母顺序，通过相邻的单元格内的字母构成，其中"相邻"单元格是那些水平相邻或垂直相邻的单元格。同一个单元格内的字母不允许被重复使用。

**解题思路**：
- 回溯算法（DFS）
- 从每个格子开始尝试
- 上下左右四个方向搜索
- 使用 visited 标记已访问的格子

**参考代码**：
```java
class Solution {
    public boolean exist(char[][] board, String word) {
        int m = board.length, n = board[0].length;
        boolean[][] visited = new boolean[m][n];
        
        for (int i = 0; i < m; i++) {
            for (int j = 0; j < n; j++) {
                if (dfs(board, word, i, j, 0, visited)) {
                    return true;
                }
            }
        }
        return false;
    }
    
    private boolean dfs(char[][] board, String word, int i, int j, int index, boolean[][] visited) {
        // 边界检查和字符匹配检查
        if (i < 0 || i >= board.length || j < 0 || j >= board[0].length 
            || visited[i][j] || board[i][j] != word.charAt(index)) {
            return false;
        }
        
        // 找到最后一个字符
        if (index == word.length() - 1) {
            return true;
        }
        
        visited[i][j] = true;
        
        // 上下左右四个方向
        boolean found = dfs(board, word, i-1, j, index+1, visited) ||
                       dfs(board, word, i+1, j, index+1, visited) ||
                       dfs(board, word, i, j-1, index+1, visited) ||
                       dfs(board, word, i, j+1, index+1, visited);
        
        visited[i][j] = false; // 回溯
        return found;
    }
}
```

**复杂度分析**：
- 时间复杂度：O(m × n × 4^L)，L 为单词长度
- 空间复杂度：O(m × n)，visited 数组和递归栈
