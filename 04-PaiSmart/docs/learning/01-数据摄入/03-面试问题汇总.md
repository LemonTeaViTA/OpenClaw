## Q1: 为什么要分片上传？

**分片上传解决了4个核心问题：**

| 问题 | 解决方案 |
|------|----------|
| 大文件HTTP超时 | 每个分片5MB，单次请求只需几秒 |
| 网络中断需重传整个文件 | 只需重传失败的分片 |
| 一次性加载大文件到内存 | 每次只处理5MB，内存可控 |
| 无法并发上传 | 多个分片可以并行上传 |

---

## Q2: 为什么选择5MB作为分片大小？

**根本原因：MinIO/S3协议的硬性限制**

```
S3协议规定：
- 最小分片大小：5MB（最后一个分片除外）
- 违反会报错：EntityTooSmall
- 这不是设计选择，而是技术约束
```

**5MB是满足MinIO要求的最小值，同时也是最优值：**
- 单片上传时间约4-8秒（4G网络），不易超时
- 服务器内存占用合理（100并发 ≈ 500MB）
- 断点续传粒度合理，失败重传代价小
- 业界通用标准

**代码中的问题：5MB是硬编码的！**
```java
int chunkSize = 5 * 1024 * 1024;  // ← 硬编码，不灵活
// 更好的方式：从配置文件读取
@Value("${file.upload.chunk-size:5242880}")
private int chunkSize;
```

## Q3: 如何实现断点续传？

**三个关键步骤：**

```
1. 前端计算文件MD5（唯一标识文件）
    ↓
2. 查询上传状态：GET /api/v1/upload/status?file_md5=abc123
   返回：{uploaded: [0,1,2,5], progress: 40.0}
    ↓
3. 只上传缺失的分片：[3,4,6,7,8,9]
```

**Redis Bitmap记录已上传的分片：**
```
文件有10个分片，已上传0,1,2,5：
位图：1110010000
      ↑↑↑  ↑
      012  5  ← 已上传
```

## Q4: 为什么用Redis Bitmap而不是List？

**内存对比（1000个分片）：**

| 存储方式 | 内存占用 | 查询方式 |
|---------|---------|----------|
| List | ~5KB | O(n)线性查找 |
| Set | ~4KB | O(1)但内存大 |
| **Bitmap** | **~125字节** | **O(1)位运算** |

**节省40倍内存！**

**一次性读取优化：**
```java
// 1次Redis请求获取所有分片状态，比逐个查询快1000倍
byte[] bitmapData = redisTemplate.execute(
    connection -> connection.get(key.getBytes())
);
```

## Q5: 三层存储各自的职责是什么？

```
Redis（Bitmap）
├─ 存什么：哪些分片已上传（0/1状态）
├─ 优点：极快，O(1)查询
└─ 缺点：可能丢失（缓存特性）

MySQL（两张表）
├─ file_upload：文件信息+权限+状态
├─ chunk_info：分片路径+MD5
├─ 优点：持久化，支持复杂查询
└─ 缺点：相对较慢

MinIO（对象存储）
├─ 存什么：实际的文件二进制数据
├─ 优点：专为文件存储设计，高可用
└─ 缺点：不支持查询
```

**为什么需要三层而不是一层？**
- 只用Redis：重启后数据丢失
- 只用MySQL：存大文件性能差
- 只用MinIO：无法快速查询状态
- **三层配合：各司其职，性能最优**

## Q6: Redis缓存失效了怎么办？

**系统设计了自动恢复机制：**

```
Redis失效 → 查询状态时Redis返回空
    ↓
从MySQL的chunk_info表恢复
SELECT * FROM chunk_info WHERE file_md5 = ?
    ↓
重建Redis Bitmap缓存
    ↓
用户无感知
```

**三层验证保证容错：**
- Redis说已上传 + MySQL有记录 + MinIO有文件 → 完全跳过
- Redis说已上传 + MySQL有记录 + MinIO文件丢失 → 重新上传
- Redis被清空 → 从MySQL + MinIO重建状态

## Q7: 如何处理高并发上传？

**四层保障：**

```
1. Kafka异步处理（核心）
   上传完成 → 立即返回 → Kafka队列 → 后台慢慢处理
   解耦上传和文档处理，削峰填谷

2. Redis缓存（减少DB压力）
   分片状态走Redis，不查MySQL
   QPS可达10万+

3. MinIO分布式存储
   多节点负载均衡，水平扩展

4. 分片机制本身
   每个请求只处理5MB，内存可控
```

## Q8: 两个用户同时上传同一分片会怎样？

**答：不会出问题，原因是幂等性设计**

```
用户A和用户B同时上传同一个分片：

1. 两个请求都通过三层验证（都说未上传）
2. 两个请求都上传到MinIO（后者覆盖前者）
3. 由于内容相同（MD5相同），覆盖没有问题
4. Redis setBit是原子操作，不会冲突
5. MySQL有唯一约束，第二次insert失败但不影响业务

结果：
✓ 数据最终一致
✓ 没有数据损坏
✓ 只是浪费了一次网络请求
```

**如果要避免重复上传，可以用分布式锁：**
```java
String lockKey = "upload_lock:" + fileMd5 + ":" + chunkIndex;
Boolean acquired = redisTemplate.opsForValue()
    .setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS);
if (acquired) {
    try { uploadToMinio(file); }
    finally { redisTemplate.delete(lockKey); }
}
```

## Q9: MinIO挂了怎么办？

**答：分层处理**

```
短暂故障（几分钟）：
→ 捕获异常，返回友好错误提示
→ 前端保存上传状态，提供重试按钮
→ MinIO恢复后断点续传，无需重头开始

长时间故障：
→ MinIO高可用部署（3-4节点集群）
→ 纠删码：1-2个节点宕机服务不中断
→ 自动健康检查 + 告警通知

生产建议：
→ MinIO集群部署
→ 数据定期备份到其他存储（如阿里OSS）
→ 建立灾难恢复预案
```

## Q10: 10000+分片的超大文件如何处理？

**MinIO的composeObject限制：最多合并10000个分片**

**解决方案：分批合并**

```
场景：15000个分片的超大文件

步骤1：分批合并成中间文件
分片0-9999    → composeObject → temp_part_0
分片10000-14999 → composeObject → temp_part_1

步骤2：合并中间文件
temp_part_0 + temp_part_1 → composeObject → 最终文件

步骤3：清理中间文件
删除 temp_part_0 和 temp_part_1
```

**当前代码的不足：**
```java
// 当前代码没有处理超过10000分片的情况
// 如果文件超过 10000 × 5MB = 50GB，会报错
// 这是一个可以优化的点！
```
