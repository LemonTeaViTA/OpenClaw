# Redis 热点题库总结

## 基础概念

### 1. 什么是 Redis？
- Redis 是一个开源的、基于内存的键值对（Key-Value）存储系统
- 可用作数据库、缓存和消息中间件
- 支持多种数据结构，提供丰富的操作命令
- 单线程模型（核心网络 IO 和键值对操作）

---

## 数据类型

### 2. Redis 有哪些数据类型？
**基本数据类型：**
- **String（字符串）**：最基本类型，可存储字符串、整数、浮点数
- **List（列表）**：双向链表，支持从两端插入/弹出
- **Set（集合）**：无序不重复集合，支持交集、并集、差集
- **Hash（哈希）**：键值对集合，适合存储对象
- **ZSet（有序集合）**：带分数的集合，按分数排序

**特殊数据类型：**
- **Bitmap**：位图，适合统计场景
- **HyperLogLog**：基数统计，占用空间极小
- **Geospatial**：地理位置索引
- **Stream**：流数据结构，支持消息队列

---

## 性能优势

### 3. Redis 为什么快？
1. **纯内存操作**：数据存储在内存中，读写速度极快
2. **单线程模型**：避免多线程上下文切换和锁竞争
3. **IO 多路复用**：使用 epoll/kqueue 实现高效网络 IO
4. **高效数据结构**：底层优化（如 SDS、跳表、压缩列表）
5. **命令简单**：大部分操作是 O(1) 或 O(log N)

---

## 持久化

### 4. Redis 的持久化方式有哪些？
**RDB（Redis Database）**
- 快照方式，定期将内存数据保存到磁盘
- 优点：文件紧凑，恢复快，适合备份
- 缺点：可能丢失最后一次快照后的数据

**AOF（Append Only File）**
- 记录每次写操作，重启时重放命令
- 优点：数据更安全，可配置每秒/每次写入
- 缺点：文件较大，恢复速度慢

### 5. Redis 4.0 混合持久化
- 结合 RDB 和 AOF 的优点
- AOF 文件前半部分是 RDB 格式快照，后半部分是增量 AOF 日志
- 优点：恢复速度快 + 数据丢失少
- 配置：`aof-use-rdb-preamble yes`

## 缓存问题

### 6. 什么是缓存击穿？
**定义**：热点 key 过期瞬间，大量请求直接打到数据库

**解决方案：**
1. **互斥锁**：只让一个线程查 DB 并回写缓存
2. **永不过期**：逻辑上不过期，后台异步更新
3. **布隆过滤器预判**：提前过滤不存在的 key

**相关概念：**
- **缓存穿透**：查询不存在的数据，请求直达 DB
- **缓存雪崩**：大量 key 同时过期，DB 压力激增

### 7. 布隆过滤器（Bloom Filter）
- 概率型数据结构，判断元素是否存在
- **优点**：空间效率极高，查询时间 O(k)
- **缺点**：有误判率（可能把不存在说成存在），不支持删除
- **应用**：缓存穿透防护、URL 去重、黑名单

---

## 数据一致性

### 8. 如何保证缓存和数据库的数据一致性？
**常见方案：**
1. **先更新 DB，再删除缓存**（推荐）
   - 延时双删：删缓存 → 更新 DB → 休眠 → 再删缓存
2. **先删除缓存，再更新 DB**
   - 可能导致旧数据回写缓存
3. **读写锁**
   - 读缓存不加锁，写操作加锁
4. **Canal 监听 Binlog**
   - 异步更新缓存，最终一致性

**选择策略：**
- 强一致场景：直接读 DB，不用缓存
- 最终一致场景：容忍短暂不一致，异步同步

---

## 内存管理

### 9. Redis 有哪些内存淘汰策略？
**6 种策略：**
1. **noeviction**：不淘汰，写满后报错（默认）
2. **allkeys-lru**：所有 key 中淘汰最近最少使用
3. **volatile-lru**：仅淘汰设置了过期时间的 key
4. **allkeys-random**：随机淘汰任意 key
5. **volatile-random**：随机淘汰带过期时间的 key
6. **volatile-ttl**：淘汰即将过期的 key

**配置：** `maxmemory-policy`

## 事务与锁

### 10. Redis 支持事务吗？
**支持，但有局限性：**
- **命令**：`MULTI`、`EXEC`、`DISCARD`、`WATCH`
- **特点**：
  - 不保证原子性（中间出错仍执行后续命令）
  - 无隔离级别（其他客户端可看到中间状态）
  - 不支持回滚
- **适用场景**：批量执行多个命令，减少网络 RTT

**真正原子操作：**
- 单命令天然原子
- Lua 脚本（Redis 保证脚本执行原子性）

### 11. Redis 能实现分布式锁吗？
**能，常见方案：**

**基础实现：**
```
SET lock_key unique_value NX EX 30
```
- NX：不存在才设置
- EX：30 秒过期时间

**完善方案：**
1. **RedLock 算法**：多 Redis 实例投票
2. **Redisson 框架**：自动续期（WatchDog 机制）
3. **释放锁校验**：Lua 脚本确保只释放自己的锁

**注意事项：**
- 锁超时时间要合理
- 客户端时钟跳变可能影响锁安全性

## 底层数据结构

### 12. Redis 有哪些底层数据结构？
| 数据类型 | 底层实现 |
|---------|---------|
| String | SDS（简单动态字符串） |
| List | 快速链表（3.2+ 用压缩列表） |
| Set | 哈希表 或 整数集合 |
| Hash | 哈希表 或 压缩列表 |
| ZSet | 跳表 + 哈希表 或 压缩列表 |

**特殊结构：**
- **压缩列表（ZipList）**：紧凑内存，适合小数据
- **快速列表（QuickList）**：链表 + 压缩列表结合
- **整数集合（IntSet）**：有序整数数组

### 13. 跳表（SkipList）
- **定义**：多层有序链表，上层是下层的索引
- **特点**：
  - 查询时间复杂度：O(log N)
  - 支持范围查询（优于平衡树）
  - 实现简单，并发友好
- **应用**：Redis ZSet 的底层实现
- **对比平衡树**：
  - 实现更简单
  - 范围查询更高效
  - 无需旋转调整
