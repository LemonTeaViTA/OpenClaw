### 题 2 - CMS 4 个阶段 ✅ 5/5
**问题**：CMS 的 4 个阶段及 STW 时机？

**用户回答**：初始标记、并发标记、重新标记、并发清除。初始标记和重新标记需要 STW，并发标记和并发清除是并发执行的。

**点评**：完全正确！上次 0/5 → 今天 5/5，进步 +5 分！

**详细对比表**：
| 阶段 | 是否 STW | 主要工作 |
|------|---------|---------|
| 初始标记 | ✅ 需要 | 标记 GC Roots 直接关联的对象 |
| 并发标记 | ❌ 并发 | 遍历整个对象图 |
| 重新标记 | ✅ 需要 | 修正并发期间变动的对象 |
| 并发清除 | ❌ 并发 | 清理未标记的对象 |

### 题 3 - CMS vs G1 对比 ✅ 4.5/5
**问题**：G1 相比 CMS 的 3 个核心优势？什么场景选择 G1？

**用户回答**：可预测停顿时间、不会产生碎片、适用于大堆（>4G），小于 4G 用 CMS。

**点评**：核心点都答对了！上次 0/5 → 今天 4.5/5，进步 +4.5 分！

**G1 三大优势**：
1. 可预测停顿（用户设定目标）
2. 无内存碎片（标记 - 整理）
3. 大堆友好（≥4G，推荐 6G+）

### 题 4 - 四种引用类型 ⚠️ 4/5
**问题**：4 种引用及弱引用 vs 虚引用的区别？

**用户回答**：强引用、软引用、弱引用、虚引用。强引用不回收，软引用内存不足回收，弱引用 GC 就回收，虚引用随时回收。

**点评**：基本正确，但虚引用的核心特点没答出来（永远无法获取对象，只能收通知）。

**关键区别**：
- 弱引用：可以通过 `get()` 获取对象，GC 发生时立即回收
- 虚引用：**永远无法获取对象**，必须配合 ReferenceQueue，只能收到回收通知

### 题 5 - 收集器搭配原则 ⚠️ 3.5/5
**问题**：Serial/ParNew 的搭配及原因？

**用户回答**：Serial 配 SerialOld，ParNew 配 CMS，Parallel Scavenge 配 Parallel Old。原因是"不是同一个系列"。

**点评**：搭配关系答对了，但原因理解不够深入（线程数匹配、设计目标一致、代际兼容）。

**深层原因**：
1. 线程数必须匹配（单线程配单线程，多线程配多线程）
2. 设计目标一致（低延迟配低延迟，吞吐量配吞吐量）
3. 代际兼容（协同工作）

### 题 6 - GC 日志分析 ✅ 4/5
**问题**：解读 GC 日志 `[GC (Allocation Failure) [PSYoungGen: 2048K->128K(2560K)] 3072K->1152K(9216K), 0.0053421 secs]`

**用户回答**：新生代回收前 2048K→回收后 128K，容量 2560K；堆回收前 3072K→回收后 1152K，容量 9216K；耗时 0.0053421 秒。

**点评**：基本全对！上次 2/5 → 今天 4/5，进步 +2 分！

**完整解读**：
- PSYoungGen = Parallel Scavenge 新生代收集器
- 新生代回收率 93.75%
- GC 耗时 5.34 毫秒（Young GC）

### 题 7 - 收集器选择场景 ⚠️ 3.5/5
**问题**：3 个场景选择收集器并说明理由

**用户回答**：
- 场景 A（8G + 低延迟）→ G1 ✅
- 场景 B（2G + 吞吐量）→ Parallel ✅
- 场景 C（4G + 批处理）→ ParNew+CMS ⚠️

**点评**：场景 A 和 B 完全正确！场景 C 不太合适（批处理对停顿不敏感，应该用 Parallel 追求吞吐量）。

**决策树**：
1. 堆大小？≥6G → G1；<6G → 继续判断
2. 关注什么？低延迟 → CMS；高吞吐 → Parallel
3. 特殊场景？批处理 → Parallel（不关心停顿）
