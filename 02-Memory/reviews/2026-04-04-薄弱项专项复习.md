#### 题 4：CAS 原理是什么？什么是 ABA 问题？如何解决？

**考点**：CAS 原理 + ABA 问题

**问题**：
1. CAS 的三要素是什么？
2. CAS 的操作流程是怎样的？
3. 什么是 ABA 问题？
4. 如何解决 ABA 问题？

<details>
<summary>点击查看答案</summary>

**答案**：

**CAS = Compare And Swap（比较并交换）**

**三要素**：
- **内存值 V**：内存中的实际值
- **预期值 A**：线程认为的当前值
- **新值 B**：线程想更新的新值

**操作流程**：
```
只有当 V == A 时，才将 V 更新为 B
否则，不更新，返回失败
```

**代码示例**：
```java
AtomicInteger atomicInt = new AtomicInteger(0);

// CAS 操作：如果当前值是 0，就更新为 1
atomicInt.compareAndSet(0, 1);  // true，更新成功

atomicInt.compareAndSet(0, 1);  // false，当前值已是 1，失败
```

**底层实现**：CPU 指令 cmpxchg（硬件级原子操作）

---

**ABA 问题**：

**场景**：
```
线程 1 读取内存值为 A
线程 2 将 A → B → A（改回原值）
线程 1 执行 CAS，发现还是 A，认为没变过 → 错误地成功
```

**问题**：值虽然回到 A，但**中间被修改过**！

**解决方案：版本号机制**
```java
AtomicStampedReference<Integer> ref = 
    new AtomicStampedReference<>(0, 0);  // 值 + 版本号

// CAS 时同时检查值和版本号
ref.compareAndSet(0, 1, 0, 1);  // 值 0→1，版本 0→1
```

**记忆口诀**：CAS 比换，ABA 版本号
</details>

#### 题 5：如何保证线程安全？有哪些实现方式？

**考点**：线程安全实现方式

**问题**：
1. 什么是线程安全？
2. 保证线程安全的三大实现方式是什么？
3. volatile 能单独保证线程安全吗？

<details>
<summary>点击查看答案</summary>

**答案**：

**线程安全**：多线程访问共享变量时，不添加额外同步措施也能保证正确性

**三大实现方式**：

| 方式 | 说明 | 示例 |
|------|------|------|
| **不可变** | 对象创建后状态不能修改 | String、final 基本类型 |
| **同步** | 通过锁或 CAS 保证原子性 | synchronized、Lock、AtomicXXX |
| **线程封闭** | 数据只在线程内部访问，不共享 | 栈封闭、ThreadLocal |

**volatile 的定位**：
- ❌ **不能单独保证线程安全**（因为不能保证原子性）
- ✅ 可以配合其他机制使用（如双重检查锁 DCL）

**记忆口诀**：不变封同（不可变、封闭、同步）
</details>

#### 题 7：死锁产生的 4 个必要条件是什么？如何避免死锁？

**考点**：死锁 4 条件 + 避免方法

**问题**：
1. 死锁产生的 4 个必要条件是什么？
2. 如何避免死锁？（至少说出 2 种方法）

<details>
<summary>点击查看答案</summary>

**答案**：

**死锁 4 个必要条件（必须同时满足）**：

| 条件 | 说明 |
|------|------|
| **互斥条件** | 资源一次只能被一个线程占用 |
| **请求与保持条件** | 线程已持有资源，又申请新资源，但不释放已持有的 |
| **不可剥夺条件** | 线程已获得的资源，不能被强行剥夺，只能自己主动释放 |
| **循环等待条件** | 线程之间形成环形等待链：A 等 B，B 等 C，C 等 A |

**避免死锁的方法（破坏任一条件）**：

| 破坏条件 | 方法 | 示例 |
|---------|------|------|
| **破坏互斥** | 用 CAS/原子类替代锁 | AtomicInteger |
| **破坏请求与保持** | 一次性申请所有资源 | 同时获取 lockA 和 lockB |
| **破坏不可剥夺** | 设置超时，获取失败就释放 | tryLock(100ms) |
| **破坏循环等待**（最常用） | **按顺序申请资源** | 线程 A 和 B 都先 lockA 后 lockB |

**记忆口诀**：互请不可循（互斥、请求、不可剥夺、循环）
</details>

### 二、集合框架 & 并发编程（3 题）

#### 题 1：阻塞队列有哪些？SynchronousQueue 和 LinkedBlockingQueue 的区别？

**考点**：阻塞队列类型 + 区别

**问题**：
1. 线程池常用的阻塞队列有哪些？
2. SynchronousQueue 的特点是什么？
3. LinkedBlockingQueue 的特点是什么？

<details>
<summary>点击查看答案</summary>

**答案**：

**常用的五种阻塞队列**：

| 队列 | 特点 | 使用场景 |
|------|------|---------|
| **ArrayBlockingQueue** | 有界，基于数组，FIFO | 固定大小的线程池 |
| **LinkedBlockingQueue** | 可无界，基于链表，吞吐量高 | 消息队列、任务队列 |
| **PriorityBlockingQueue** | 无界，支持优先级排序 | 紧急任务优先处理 |
| **DelayQueue** | 无界，延迟执行 | 定时任务、缓存过期 |
| **SynchronousQueue** | 不存储元素，直接交接 | CachedThreadPool |

**SynchronousQueue vs LinkedBlockingQueue**：

| 特性 | SynchronousQueue | LinkedBlockingQueue |
|------|-----------------|---------------------|
| **容量** | 0（不存储元素） | 可指定，默认 Integer.MAX_VALUE |
| **存储** | 不存储，直接交接 | 基于链表存储 |
| **阻塞** | 插入和移除互相等待 | 满时插入阻塞，空时移除阻塞 |
| **使用场景** | Executors.newCachedThreadPool() | 自定义线程池、消息队列 |

**记忆要点**：
- SynchronousQueue = 直接传递，不存储
- LinkedBlockingQueue = 链表存储，可无界
</details>

#### 题 2：哈希冲突的解决方法有哪些？

**考点**：哈希冲突解决

**问题**：
1. 什么是哈希冲突？
2. 哈希冲突的解决方法有哪些？
3. HashMap 用的是哪种方法？

<details>
<summary>点击查看答案</summary>

**答案**：

**哈希冲突**：多个键经过哈希处理后得到相同的索引位置

**解决方法**：

| 方法 | 说明 | 示例 |
|------|------|------|
| **拉链法（链地址法）** | 用链表串联冲突元素 | **HashMap 采用** |
| **开放寻址法** | 寻找下一个空槽 | 线性探测、二次探测、双重哈希 |
| **再哈希法** | 使用多套哈希算法 | 较少使用 |
| **建立公共溢出区** | 冲突元素放入溢出区 | 较少使用 |

**HashMap 的拉链法优化**：
- 链表长度 < 8：使用链表
- 链表长度 ≥ 8 且数组长度 ≥ 64：转为红黑树
- 目的：提高查询效率（O(n) → O(logn)）

**记忆口诀**：HashMap 用拉链，太长变红黑
</details>

#### 题 3：ConcurrentHashMap 的原理是什么？JDK7 和 JDK8 的区别？

**考点**：ConcurrentHashMap 实现原理

**问题**：
1. ConcurrentHashMap 如何保证线程安全？
2. JDK7 和 JDK8 的实现有什么区别？
3. 为什么 JDK8 性能更好？

<details>
<summary>点击查看答案</summary>

**答案**：

**JDK7 实现**：
- **分段锁（Segment）**：整个 Map 分为若干段，每段独立加锁
- **锁类型**：ReentrantLock
- **数据结构**：数组 + 链表
- **并发度**：取决于 Segment 数量（默认 16）

**JDK8 实现**：
- **桶锁（Bucket）**：更细粒度的锁，只锁当前桶
- **锁类型**：CAS + synchronized
- **数据结构**：数组 + 链表 + 红黑树
- **并发度**：取决于数组长度（默认 16，可扩容）

**JDK8 优化点**：

| 优化 | JDK7 | JDK8 |
|------|------|------|
| **锁粒度** | 段锁（Segment） | 桶锁（Node） |
| **锁实现** | ReentrantLock | CAS + synchronized |
| **数据结构** | 数组 + 链表 | 数组 + 链表 + 红黑树 |
| **hash 计算** | 二次哈希 | spread 方法（高位异或低位） |
| **性能** | 较低 | 更高 |

**put 流程（JDK8）**：
1. 计算 hash，定位桶位置
2. 桶为空：CAS 插入
3. CAS 失败：synchronized 锁当前桶
4. 链表长度≥8：转红黑树
5. 检查扩容

**记忆口诀**：7 分段，8 桶锁，CAS 加同步红黑树
</details>
