## L1 基础题（项目理解）⭐⭐⭐

### Q1: 派聪明的核心功能是什么？

**考察点**：表达能力、项目熟悉度

**参考答案**：
```
派聪明是一个基于 RAG 的企业级知识库系统，核心功能有 3 个：

1. 知识摄入：支持多种格式文档上传（PDF/Word/Excel 等），自动解析和向量化
2. 智能检索：混合检索（向量 KNN + 全文 BM25），支持权限过滤
3. RAG 对话：基于检索结果生成回答，支持多轮对话

技术栈：Spring Boot + Elasticsearch + Kafka + MinIO + DeepSeek
```

---

### Q2: 用户上传一个文件的完整流程是什么？

**考察点**：系统理解、数据流

**参考答案**：
```
1. UploadController 接收 HTTP 请求（文件分片）
2. UploadService 存储到 MinIO（分片上传）
3. Redis Bitmap 记录上传状态（断点续传）
4. MySQL 记录文件元数据
5. 合并分片（MinIO composeObject）
6. 发送 Kafka 消息（异步处理）
7. FileProcessingConsumer 消费消息
8. ParseService 解析文档（Apache Tika + HanLP 切分）
9. VectorizationService 向量化（通义千问 API）
10. 存储到 Elasticsearch（供检索使用）
```

---

### Q3: 为什么要用 Kafka 做异步处理？

**考察点**：技术选型理解

**参考答案**：
```
Kafka 解决了 3 个问题：

1. 削峰填谷：上传高峰时，Kafka 缓冲，后端按能力消费
2. 解耦：上传服务不关心处理逻辑，发送消息即可
3. 失败重试：消费失败可重试，死信队列兜底

如果不用 Kafka：
- 同步处理：用户上传后需等待解析+向量化（可能几分钟）
- 体验差：HTTP 请求超时风险
- 耦合：上传服务依赖解析服务
```

## L2 技术题（技术选型）⭐⭐⭐⭐

### Q4: 为什么文件分片大小选择 5MB？

**考察点**：技术细节、原理理解

**参考答案**：
```
根本原因：MinIO/S3 协议的硬性限制

S3 协议规定：
- 最小分片大小：5MB（最后一个分片除外）
- 违反会报错：EntityTooSmall

同时 5MB 也是最优值：
- 单片上传时间约 4-8 秒（4G 网络），不易超时
- 服务器内存占用合理（100 并发 ≈ 500MB）
- 断点续传粒度合理，失败重传代价小
```

**代码问题**：
```java
// 当前代码：硬编码
int chunkSize = 5 * 1024 * 1024;

// 更好的方式：从配置文件读取
@Value("${file.upload.chunk-size:5242880}")
private int chunkSize;
```

### Q5: 为什么用 Redis Bitmap 而不是 List 记录分片状态？

**考察点**：数据结构选型、性能优化

**参考答案**：
```
内存对比（1000 个分片）：

| 存储方式 | 内存占用 | 查询方式 |
|---------|---------|----------|
| List | ~5KB | O(n) 线性查找 |
| Set | ~4KB | O(1) 但内存大 |
| Bitmap | ~125 字节 | O(1) 位运算 |

Bitmap 节省 40 倍内存！

而且支持批量操作：
- 设置已上传：SETBIT file:md5 chunkIndex 1
- 查询已上传：GETBIT file:md5 chunkIndex
- 批量查询：GETRANGE 一次性读取所有位
```

### Q6: 混合检索为什么同时用 KNN 和 BM25？

**考察点**：检索原理、技术深度

**参考答案**：
```
KNN 和 BM25 互补：

| 检索方式 | 原理 | 优势 | 劣势 |
|---------|------|------|------|
| KNN（向量） | 语义相似度 | 找"意思相近"的内容 | 精确匹配差 |
| BM25（全文） | 词频统计 | 精确匹配专有名词 | 语义理解差 |

示例：
- 用户搜"怎么请假"
- KNN：找到"请假流程"（语义相近）
- BM25：找到"请假申请表"（精确匹配）

融合策略：RRF（Reciprocal Rank Fusion）
```
score = 1 / (60 + rank)
```

---

### Q7: 向量化为什么选择批处理？

**考察点**：性能优化、API 限制

**参考答案**：
```
原因：

1. API 限制：通义千问 embedding 每次最多 10 条
2. 减少网络开销：10 条一起发比单独发 10 次快
3. 提升吞吐量：批处理减少 API 调用次数

代码实现：
```java
// 累积到 10 条或超时 1 秒，批量发送
if (batch.size() >= 10 || System.currentTimeMillis() - startTime > 1000) {
    List<float[]> embeddings = embeddingClient.embedDocuments(batch);
    // 批量写入 ES
}
```

**优化点**：
- 增加重试机制（网络波动）
- 失败降级（记录日志，稍后重试）
```

## L3 场景题（扩展场景）⭐⭐⭐⭐⭐

### Q8: 如果 QPS 翻 10 倍，系统哪里会先挂？怎么优化？

**考察点**：系统设计、性能瓶颈分析

**参考答案**：
```
瓶颈分析：

1. **MinIO 上传**：带宽瓶颈
   - 优化：CDN 加速、多节点负载均衡

2. **Kafka 消费**：解析速度慢
   - 优化：增加消费者数量、并行处理

3. **向量化 API**：调用频率限制
   - 优化：本地缓存（相同文本不重复调用）、批量处理

4. **Elasticsearch**：写入压力
   - 优化：bulk 批量写入、增加节点

5. **数据库**：连接数不足
   - 优化：连接池调优、读写分离

**优先级**：
1. 向量化 API（最可能限流）
2. Kafka 消费（解析慢）
3. ES 写入
```

### Q9: 如果用户上传了敏感文件（如机密文档），如何防止泄露？

**考察点**：安全设计、权限控制

**参考答案**：
```
多层防护：

1. **上传时**：
   - 权限校验（只有特定角色可上传机密）
   - 内容审计（敏感词检测）

2. **存储时**：
   - 加密存储（MinIO 服务端加密）
   - 隔离存储（机密文件单独 bucket）

3. **检索时**：
   - 权限过滤（orgTag + 用户角色）
   - 审计日志（谁搜了什么）

4. **返回时**：
   - 脱敏处理（敏感信息打码）
   - 水印（截图可追溯）

**代码实现**：
```java
// HybridSearchService 权限过滤
BoolQueryBuilder query = QueryBuilders.boolQuery()
    .must(knnQuery)
    .filter(QueryBuilders.termQuery("orgTag", userOrgTag))  // 组织隔离
    .filter(QueryBuilders.termsQuery("permission", userRoles));  // 角色过滤
```

### Q10: 如果 DeepSeek API 挂了，如何保证服务可用？

**考察点**：高可用、降级策略

**参考答案**：
```
降级方案：

1. **多模型备份**：
   - 主：DeepSeek
   - 备：通义千问、文心一言
   - 自动切换（主模型失败率>10% 切备胎）

2. **缓存答案**：
   - 相似问法缓存（向量相似度>95% 直接返回）
   - 热门问题预计算

3. **本地模型**：
   - 小模型兜底（如 ChatGLM-6B）
   - 效果差但可用

4. **友好提示**：
   - "AI 服务暂时不可用，请稍后再试"
   - 提供人工客服入口

**代码实现**：
```java
try {
    return deepSeekClient.chat(prompt);
} catch (Exception e) {
    // 降级到备用模型
    return qwenClient.chat(prompt);
}
```

## L4 难点题（攻克难点）⭐⭐⭐⭐⭐

### Q11: 项目中遇到的最大技术挑战是什么？如何解决的？

**考察点**：问题解决、成长反思

**参考方向**：
```
可选答案（选 1 个深入讲）：

1. **大文件解析 OOM**：
   - 问题：一次性加载 100MB PDF 到内存 → OOM
   - 解决：流式处理（1MB chunk）、分批次向量化
   - 结果：内存从 2GB 降到 200MB

2. **检索结果不精准**：
   - 问题：KNN 检索返回无关内容
   - 解决：混合检索（KNN+BM25）+ RRF 融合
   - 结果：准确率从 60% 提升到 85%

3. **高并发上传失败率高**：
   - 问题：100 并发上传，失败率 30%
   - 解决：Redis Bitmap 原子操作 + 重试机制
   - 结果：失败率降到 1% 以下
```

### Q12: 如果让你重新设计派聪明，你会改进什么？

**考察点**：反思能力、架构思维

**参考方向**：
```
可选答案：

1. **微服务拆分**：
   - 当前：单体应用（上传、解析、检索耦合）
   - 改进：拆分为上传服务、解析服务、检索服务
   - 好处：独立扩展、故障隔离

2. **流式架构**：
   - 当前：Kafka 批处理
   - 改进：Flink 实时流处理
   - 好处：更低延迟、Exactly-Once 语义

3. **多租户支持**：
   - 当前：单租户（orgTag 简单隔离）
   - 改进：独立数据库/索引 per tenant
   - 好处：数据隔离更彻底、可定制化

4. **监控告警**：
   - 当前：基础日志
   - 改进：Prometheus + Grafana + 告警
   - 好处：问题及时发现
```
