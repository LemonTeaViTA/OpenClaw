## 📖 核心概念

### 什么是 RAG？

**RAG = Retrieval-Augmented Generation（检索增强生成）**

RAG 是一种结合**检索**和**生成**的 AI 技术：
1. 先从外部知识库检索相关信息
2. 再将检索结果作为上下文提供给 LLM
3. LLM 基于检索结果生成答案

**核心思想**：让 AI"开卷考试"，而不是"闭卷考试"。

---

## 🔍 为什么需要 RAG？

### LLM 的局限性

| 问题 | 说明 | RAG 解决方案 |
|------|------|------------|
| **知识过时** | 训练数据有截止日期 | 检索最新资料 |
| **幻觉问题** | 可能编造虚假信息 | 基于真实资料生成 |
| **领域知识缺乏** | 缺乏特定领域知识 | 检索专业文档 |
| **上下文限制** | 上下文窗口有限 | 只检索相关内容 |

## 🏗️ RAG 架构

### 核心组件

```
用户问题
    ↓
┌─────────────────┐
│  问题 Embedding  │  ← 将问题转换为向量
└─────────────────┘
    ↓
┌─────────────────┐
│  向量数据库检索  │  ← 查找相似文档
└─────────────────┘
    ↓
┌─────────────────┐
│  检索结果排序    │  ← 选择最相关的 Top-K
└─────────────────┘
    ↓
┌─────────────────┐
│  LLM 生成答案    │  ← 基于检索结果生成
└─────────────────┘
    ↓
最终答案
```

### 技术栈

| 组件 | 常用工具 |
|------|---------|
| **Embedding 模型** | text-embedding-3-small、BGE、M3E |
| **向量数据库** | Milvus、Chroma、Pinecone、Elasticsearch |
| **LLM** | GPT-4、Claude、文心一言 |
| **编排框架** | LangChain、LlamaIndex |

## 📊 RAG 工作流程

### 步骤 1：文档预处理

```
原始文档 → 分块（Chunking） → Embedding → 存入向量数据库
```

**分块策略**：
- 按段落分块（每块 500-1000 Token）
- 按语义分块（保持语义完整）
- 重叠分块（相邻块重叠 10-20%）

### 步骤 2：检索

```
用户问题 → Embedding → 向量相似度计算 → Top-K 结果
```

**相似度算法**：
- 余弦相似度（Cosine Similarity）
- 欧氏距离（Euclidean Distance）
- 点积（Dot Product）

### 步骤 3：生成

```
Prompt = 系统指令 + 检索结果 + 用户问题
LLM → 基于 Prompt 生成答案
```

**Prompt 模板**：
```
你是一个专业助手。请根据以下资料回答问题：

【相关资料】
{检索结果 1}
{检索结果 2}
{检索结果 3}

【问题】
{用户问题}

【回答】
```

## 🎯 RAG 的应用场景

### 场景 1：企业知识库问答

**案例**：派聪明（PaiSmart）

- **知识库**：企业文档、产品手册、技术文档
- **检索**：员工提问 → 检索相关文档
- **生成**：基于文档生成答案

**优势**：
- ✅ 答案准确（基于真实文档）
- ✅ 知识实时更新
- ✅ 减少幻觉

---

### 场景 2：客服机器人

**案例**：电商客服

- **知识库**：商品详情、售后政策、物流信息
- **检索**：用户咨询 → 检索相关政策
- **生成**：生成个性化回复

**优势**：
- ✅ 7×24 小时在线
- ✅ 回答一致性强
- ✅ 降低人工成本

---

### 场景 3：法律/医疗问答

**案例**：法律咨询助手

- **知识库**：法律法规、案例库、司法解释
- **检索**：法律问题 → 检索相关法条
- **生成**：生成法律建议

**优势**：
- ✅ 答案有法条依据
- ✅ 可追溯来源
- ✅ 降低专业门槛

## ⚠️ RAG 的挑战

### 挑战 1：检索质量

**问题**：
- 检索结果不相关
- 遗漏关键信息
- 检索结果过多/过少

**解决方案**：
- 优化分块策略
- 调整 Top-K 数量
- 使用混合检索（关键词 + 向量）

---

### 挑战 2：上下文长度

**问题**：
- 检索结果太多，超出 LLM 上下文窗口
- 关键信息被截断

**解决方案**：
- 压缩检索结果（只保留关键句）
- 分层检索（先粗选再精选）
- 使用长上下文模型（如 Claude 200K）

---

### 挑战 3：实时性

**问题**：
- 向量数据库更新延迟
- 新文档无法立即检索

**解决方案**：
- 增量更新（只更新变更部分）
- 实时索引（文档上传即索引）
- 混合检索（新文档走关键词检索）

## 📝 面试原题

### 题 1：什么是 RAG？

**参考答案**：
> RAG = Retrieval-Augmented Generation（检索增强生成）。它结合检索和生成：先从外部知识库检索相关信息，再将检索结果作为上下文提供给 LLM，让 AI"开卷考试"。核心优势是减少幻觉、知识可实时更新、答案可追溯来源。

**评分标准**：
- ✅ 核心定义（检索 + 生成）
- ✅ 核心思想（开卷考试）
- ✅ 优势（减少幻觉、实时更新、可追溯）

---

### 题 2：RAG 的工作流程是什么？

**参考答案**：
> 1. 文档预处理：分块 → Embedding → 存入向量数据库
> 2. 检索：问题 Embedding → 向量相似度计算 → Top-K 结果
> 3. 生成：构建 Prompt（系统指令 + 检索结果 + 问题）→ LLM 生成答案

**评分标准**：
- ✅ 三步流程（预处理、检索、生成）
- ✅ 关键技术（Embedding、向量检索、Prompt）

---

### 题 3：RAG 相比纯 LLM 有什么优势？

**参考答案**：
> 1. **减少幻觉**：基于真实资料生成，不编造信息
> 2. **知识实时更新**：检索最新资料，不受训练数据限制
> 3. **领域知识增强**：检索专业文档，弥补 LLM 领域知识不足
> 4. **答案可追溯**：可以标注答案来源，便于验证

**评分标准**：
- ✅ 至少说出 3 个优势
- ✅ 每个优势有简短解释

---

### 题 4：RAG 的挑战有哪些？

**参考答案**：
> 1. **检索质量**：可能检索到不相关内容
> 2. **上下文长度**：检索结果可能超出 LLM 上下文窗口
> 3. **实时性**：向量数据库更新可能有延迟
> 4. **分块策略**：分块大小影响检索效果

**评分标准**：
- ✅ 至少说出 2 个挑战
- ✅ 每个挑战有解决方案思路
